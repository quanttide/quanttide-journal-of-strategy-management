我最近有了一个新的想法：在战略上，我们可以尽可能利用开源社区的方式，让开源社区的成果为我们所用。

我们有一个很大的优势，就是具备开源贡献能力，因此可以很好地与开发者保持关系。这样一来，我们其实可以成为开源社区中具备一定领导力的角色，因为相比于很多个人开发者，我们的场景更多，具备组织大家的能力，我觉得这一点非常重要。

此外，有很多很不错的开源项目，我们可以与他们互通有无，这能带来以下好处：
1. 提高我们的生产力；
2. 为我们自己的社区做冷启动；
3. 从乙方和甲方的角度，去理解我们未来面临的甲方会是什么样子。

对我我觉得既然我们要开打币，就应该最大限度地去串联开源社区，跟开源社区合作。因为现在的研发能力决定了，像我个人去贡献一些代码已经没有什么难的了，而且这种能力很有可能被逐渐积累到团队上面。

因为我们目前不需要去造自己的编程智能体了，直接用 IS Coder 就可以实现我们原本计划的效果。这样一个比较理想的东西，放在我们的 dev 流程里面去自动化编码这个环节也是蛮合适的，所以我觉得这是一个起点。

然我选择的对象，主要是那些像我们一样有不错的想法的人。像我们可能成熟度还不够，而有一些专门做某一个项目的，可能已经有了一定的规模，那么就可以用起来。
 
因为我们的想法是其实有一点复杂，一时半会做不出来，这个倒也正常。我们可以多从社区吸收一些方法，吸收到我们项目里，我觉得这是一个很重要的事情。

我在考虑我的 Code 工具：

我想把一部分简单的功能放到 DevOps 这边，增强这边对编码能力的管理。Code 那边则不要过度纠结代码规范，而是能够把东西写出来，这会更像 S-Code。

我觉得它会更接近我对于编程智能体的需求，也就是一个更加可控、常识的 System 2，可以用来放在后台去执行任务。

然后，因为我们可以去跟他进行贡献和和 PR，所以说呢，我觉得可以通过这种方式保持一个比较良好的关系

通过生产关系去建立真正的社区关系，驱动开源社区内的互动，进而驱动增长。我觉得这是一个很重要的考虑和布局。

通过这种方式，可以更好地进行更大范围的社区合作，并且能够最大限度地利用社区成果，用开源的网络去和闭源的厂商竞争。

我觉得这是最关键的一个系统，它考验的其实是我们的组织能力。

所以理论上，我们可以有一个开源云。这个开源云其实就是用来跟社区互动，无论是我们去开源，还是参与开源，我觉得这都是一种共享的行为。那么，我就可以从现在的活动之中找到一些经验。

我觉得开源这一块的核心，其实就在于大家互相交流、互相适应。如果能找到更加一致的标准，对于我们来讲，做标准化也会比较容易一点。

对，所以“开打 B”这个战略真正的核心，就不是“开”就完了，而是我们要通过开源，去和愿意开源的人建立更广泛的同盟关系，最大限度地降低我们的成本，去争取社区的话语权和领导权。
